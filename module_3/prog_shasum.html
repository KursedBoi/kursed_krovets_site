Due: Sun April 3, 11:59pm.

You may collaborate with one or two other students on this homework if you wish, or work alone. Collaboration must be true collaboration however, which means that the work put into each problem should be roughly equal and all parties should come away understanding the solution. Here are some suggested ways of collaborating on the programming part.

  • Pair programming. Two or three of you look at the same screen and only one of you operate the keyboard. The one at the keyboard is the "driver" and the other is the "navigator". The driver explains everything they are doing as they do it and the navigator asks questions and makes suggestions. If the navigator knows how to solve the problem and the driver does not, the navigator should not dictate solutions to the driver but instead should tutor the driver to help them understand. The driver and navigator should switch roles every 10-15 minutes. Problems solved this way can then be individually submitted.
  • Code review. The members of the collaborative each try to solve the problem independently. They then take turns analyzing each other's code, asking questions trying to understand each other's algorithms and suggesting improvements. After the code reviews, each of you can then fix your code using what you learned from the reviews. Do not copy code. If the result of code review is that your code needs changes to be more like your partner's, do not copy it. Instead recreate your own variant without looking at your partner's.

The goal is to learn enough from one another so that you each can do similar problems independently in an exam situation.

If you want a collaborator but don't know people in the class, you can ask on Discord and/or use the group-finding post on Piazza.

Homework Assignment

  1. A useful command line tool on many Linux amd Mac computers is "shasum filename" which reads the contents of a file, hashes it with a cryptographic hash function, and prints the reult in hex. The collision resistance of a hash function tells you that if two files hash to the same result they are almost certainly the same file.

    Log onto a Linux or Mac system and type "man shasum" to see how it works. Then experiment by hashing some files with SHA2-256.

    Write a program using OpenSSL that takes a file name as a command line input (ie, argv[1] will hold its name) and performs a SHA2-256 hash on it. Your program should then print the 32 byte result in lower-case hex. For example, here's my program running on ecs-pa-apollo.

    ecs-pa-apollo> gcc shasum256.c -lcrypto
    ecs-pa-apollo> ./a.out shasum256.c 
    a41ebc078db87b2ce641b2d05f3bb8d892152fa235296a0edbb85739c4e04226

    Your code should output 64 lower-case hex characters followed by a newline and nothing else.

  2. Write a function divlessmod that takes a uint64_t number x and a positive int number a and returns a uint64_t value congruent to x mod 2a-1 by applying divisionless mod once. (To create an a-bit mask of 1-bits, you can use the expression ((uint64_t)1 << a) - 1 which is equivalent to 2a-1 which is represented as a 1-bits in binary).

  3. Write a function divlessmod_final that takes a uint64_t number x and a positive int number a and returns a uint64_t value equal to x mod 2a-1 but do it without using % (which is expensive) and without using any conditional computation depending on x (which, if x is a secret value, could lead to "timing attacks").

    Do this by doing one more application of divisionless mod. This could leave your result slightly greater or equal to 2a-1, which is not allowed and needs to be reduced. Logically we could handle this with if (x>=prime) x=x-prime but that conditional could lead to a timing attack and so should be avoided.

    One way to implement if (x>=prime) x=x-prime without the conditional is through clever bit-level operations. If x ≥ prime, then t = (prime-1) - x is a negative number (represented in twos'complement binary with a 1-bit in the leftmost position) and positive otherwise. The value of (int64_t)t >> 63 will thus be 64 1-bits if x ≥ prime and 64 0-bits otherwise. (A right shift of a twos-complement signed integer replicates the high bit.) You can use this resulting value as a bit mask to help solve your problem without any conditionals.

To simplify your coding tasks, your code does not need to be robust. You may assume all tests will contain legal inputs. You should test your code as you see fit for now. You will submit your solution via Mimir closer to the deadline. OpenSSL code will be tested using OpenSSL version 1.1 (the same version installed on ecs-pa-apollo).

Hints

  • The documentation I use most for OpenSSL programming are:

    • OpenSSL Wiki's EVP examples. Find the closest example and study the webpage for it.
    • OpenSSL's man pages. Make sure the OpenSSL version you are using matches the man page you look at. I usually find the man page for the function using internet search (eg, search for "openssl rand_byes").
    • BoringSSL's API reference. Google forked OpenSSL a while ago, so their API doesn't match OpenSSL's exactly, but when they do match, Google's documentation tends to be clearer.

  • The C reading from the first week has an example of reading a file.

  • %02x is the format specifier for printing a byte a two-character lower-case hex (the 0 inserts a 0 as the high nibble if the byte is less than 16).

  • You can read about compiling with OpenSSL on your own system.

  • Before you submit your code, compile it with extra command line arguments -Wall -Werror -fsanitize=address and make sure it still runs correctly. The -W options will force you to eliminate your compiler warnings. The -fsanitize option will attempt to uncover illegal memory accesses by your program.
